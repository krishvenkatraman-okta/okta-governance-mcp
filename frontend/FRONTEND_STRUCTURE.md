## Route Explanations

### Authentication Flow

```
1. User visits /login
   ↓
2. Clicks "Sign in with Okta"
   ↓
3. Redirects to /api/auth/start
   ↓
4. Server generates PKCE challenge
   ↓
5. Redirects to Okta authorize endpoint
   ↓
6. User authenticates with Okta
   ↓
7. Okta redirects to /api/auth/callback with code
   ↓
8. Server exchanges code for tokens (PKCE)
   ↓
9. Stores tokens in session
   ↓
10. Redirects to /agent
```

### Token Exchange Flow

```
After OAuth callback:

1. Server has ID token from Okta
   ↓
2. POST /api/token/id-jag
   → Exchange ID token for ID-JAG
   ↓
3. POST /api/token/access-token
   → Exchange ID-JAG for access token
   ↓
4. Access token stored in session
   ↓
5. Used for all MCP server requests
```

### MCP Integration

```
From /agent page:

1. User clicks "Load Tools"
   ↓
2. Frontend calls GET /api/mcp/tools
   ↓
3. Server retrieves access token from session
   ↓
4. Server calls MCP server with Bearer token
   ↓
5. MCP server returns filtered tools list
   ↓
6. Frontend displays tools
   ↓
7. User selects tool and clicks "Invoke"
   ↓
8. Frontend calls MCP server to execute tool
```

## Environment Variables

### Public (Client-side accessible)

These are exposed to the browser:

- `NEXT_PUBLIC_OKTA_DOMAIN` - Okta domain
- `NEXT_PUBLIC_OKTA_CLIENT_ID` - OAuth client ID
- `NEXT_PUBLIC_OKTA_CUSTOM_AUTH_SERVER` - Auth server ID
- `NEXT_PUBLIC_REDIRECT_URI` - OAuth callback URL
- `NEXT_PUBLIC_MCP_BASE_URL` - MCP backend URL

### Private (Server-side only)

These are only accessible on the server:

- `SESSION_SECRET` - Secret for encrypting session cookies
- `NODE_ENV` - Environment (development/production)
